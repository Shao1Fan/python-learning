# 第二部分：50 道高频面试题（附答案要点）

## 🅰 大模型基础（10 题）

### Q1：Transformer 中为什么要用 Scaled Dot-Product Attention？为什么除以 sqrt(d_k)？

> **答案要点**：避免 softmax 进入梯度饱和区。d_k 较大时点积方差增大，梯度极小，除以 sqrt(d_k) 将方差归一化为 1，保证梯度稳定性。

### Q2：RoPE（Rotary Position Embedding）的原理是什么？相比 Sinusoidal PE 有什么优势？

> **答案要点**：RoPE 通过旋转矩阵对 token 的 query 和 key 施加位置信息，使 Attention 天然具备相对位置感知能力。优势：① 支持外推（通过引入高频/低频旋转）；② 相对位置衰减特性；③ 与线性注意力兼容。

### Q3：什么是 GQA（Grouped Query Attention）？为什么 LLaMA 2/3 和 Qwen 2 都用它？

> **答案要点**：GQA 将 Query 分为 G 组，每组共享一个 Key/Value head，是 MHA（全头）和 MQA（单头）的折中。优势：减少 KV-Cache 显存占用（接近 MQA），同时保持接近 MHA 的效果。

### Q4：MoE 架构中 Gate 是如何工作的？如何解决负载不均衡问题？

> **答案要点**：Gate 是一个线性层 + Softmax + Top-K 选择，输出每个 token 被分配到哪些 expert 的概率。负载不均衡通过辅助 Loss 解决：① Load Balance Loss 鼓励均匀分配；② Z-loss 稳定训练。

### Q5：DeepSeek MoE 相比标准 MoE 有什么创新？

> **答案要点**：① 细粒度 Expert 切分；② Shared Expert（共享知识，减少各 expert 冗余学习）；③ Device-level Auxiliary Loss。

### Q6：什么是 KV-Cache？为什么 Prefill 阶段和 Decode 阶段的计算复杂度不同？

> **答案要点**：KV-Cache 缓存已生成 token 的 K/V 向量避免重复计算。Prefill（预填充）：并行计算所有 prompt token（计算密集型）。Decode（逐 token 生成）：逐个生成（Memory-bound），每次仅计算新 token。

### Q7：Flash Attention 是如何解决 Attention 显存瓶颈的？

> **答案要点**：核心思想：分块计算 + 在线 softmax，避免 O(N²) 显存占用。通过分块从 HBM 到 SRAM，用 tiling 方法分块计算，使时间复杂度不变但显存从 O(N²) 降到 O(N)。

### Q8：RMSNorm 和 LayerNorm 的区别是什么？为什么大模型更常用 RMSNorm？

> **答案要点**：RMSNorm 去掉 LayerNorm 中的均值中心化步骤，只做 RMS 归一化。优势：计算量减少（约 5-10%），经验上在大规模场景效果相当且更稳定。

### Q9：LLaMA 3 相比 LLaMA 2 有哪些关键改进？

> **答案要点**：① 词汇表从 32K 扩大到 128K；② 使用 GQA；③ 训练数据更大且更多代码/多语言；④ 更长的训练；⑤ 分组查询注意力在全部层都使用。

### Q10：什么是 Multi-Token Prediction（MTP）？DeepSeek 为什么用它？

> **答案要点**：MTP 在训练时要求模型同时预测未来多个位置的 token（不仅下一个）。优势：① 增加训练信号密度；② 天然有利于推理时生成更长的连贯序列；③ DeepSeek 使用 MTP 作为训练辅助目标。

## 🅱 Post-Training 与对齐（12 题）

### Q11：SFT 阶段应该用多少数据？数据质量如何保证？

> **答案要点**：通常 1K-100K 高质量指令即可（不是越多越好）。数据筛选策略：① Instruction 多样性（基于 embedding 聚类）；② Response 质量（人工标注 / LLM as Judge 打分）；③ 去重（MinHash + LSH）；④ 排除 benchmark 污染样本。

### Q12：什么是灾难性遗忘？如何缓解？

> **答案要点**：微调时模型会失去预训练阶段学习到的通用知识。缓解方法：① SFT 混合预训练数据（10-20%）；② Elastic Weight Consolidation（EWC）；③ Replay/Experience Replay；④ 使用较小学习率 + 较少 epoch。

### Q13：LoRA 的原理是什么？rank 的选择对效果有什么影响？

> **答案要点**：LoRA 将权重更新 ΔW 分解为低秩矩阵 BA（W + BA），训练时冻结 W 只更新 B/A。rank r 决定了微调的表达能力：r 太小 → 表达能力不足；r 太大 → 接近全参训练，失去 LoRA 优势。建议 r=8-64。

### Q14：RLHF 的三阶段各做什么？为什么需要这样做？

> **答案要点**：
> ① **SFT**：让模型学会指令跟随格式
> ② **Reward Model**：训练一个反映人类偏好的打分模型
> ③ **PPO RL**：以 RM 为奖励信号优化策略，同时 KL 惩罚防止模型偏离太远
> 三者缺一不可：没有 RM 无法量化偏好，没有 PPO 无法在探索中优化。

### Q15：DPO 是如何绕过显式 Reward Model 的？推导过程了解吗？

> **答案要点**：DPO 从 RLHF 的 KL-constrained 优化目标出发，推导出最优策略 π* 的 closed-form 表达式，将其代入偏好概率公式，得到直接基于偏好对的损失函数。关键公式：L = -E[log σ(β log(π_θ(y_w|x)/π_ref(y_w|x)) - β log(π_θ(y_l|x)/π_ref(y_l|x)))]

### Q16：DPO 和 PPO 各自的优缺点是什么？

> **答案要点**：
> | 维度 | PPO | DPO |
> |------|-----|-----|
> | 复杂度 | 需要 RM + Value + Actor 共 3 个模型 | 只需 1 个模型 |
> | 训练稳定 | 较难调参 | 较稳定 |
> | 探索能力 | 有探索（采样） | 只在静态偏好对上优化 |
> | 分布覆盖 | 可生成新数据 | 依赖偏好数据集质量 |

### Q17：GRPO（Group Relative Policy Optimization）的原理是什么？DeepSeek-R1 为什么要用它？

> **答案要点**：GRPO 不使用 Critic 模型（Value Function），而是对同一 prompt 采样多个 response，用组内相对优势来估计优势函数。优势：① 节省 Critic 模型的显存和训练成本；② 天然适合推理任务（一个 prompt 可多路尝试不同思路）。

### Q18：Process Reward Model（PRM）和 Outcome Reward Model（ORM）的区别？

> **答案要点**：ORM 对整个 response 给一个最终分数；PRM 对每一步（每个推理步骤）分别打分。ORM 简单但无法定位中间错误；PRM 可发现"过程对但结果错"或"过程错但结果碰对"的情况，更适合数学/推理任务。

### Q19：什么是 Rejection Sampling？在 Agent 训练中如何使用？

> **答案要点**：从当前策略采样 N 个候选 response，选得分最高的去训练。在 Agent 训练中：对同一任务让 Agent 尝试多种策略路径，保留成功完成的路径用于 SFT 或偏好学习。

### Q20：Self-Instruct 的完整流程是什么？有什么局限性？

> **答案要点**：流程：种子 Instruction → LLM 生成新 Instruction → 生成 Response → 过滤（质量/去重/多样性）→ 迭代。局限性：合成的多样性受限于种子集；指令质量无法超过原始模型的能力。

### Q21：长上下文中 Agent 的效果退化如何处理？

> **答案要点**：① 滑动窗口 / 自动摘要机制压缩记忆；② 分层的 Memory 架构（工作记忆 + 长期记忆）；③ 关键信息优先检索而不是全量 Attention；④ Agent Reflection 机制自我过滤无用历史。

### Q22：什么是 Synthetic Data RL？如何用它来训练 Agent？

> **答案要点**：使用 LLM 模拟用户和环境的交互数据，生成多样化的 Agent 任务场景和对应的最优路径。然后用这些合成数据做 SFT 或 RL。优势：① 降低人工标注成本；② 可生成大量覆盖 edge case 的场景。

## 🅲 Agent 架构与实践（12 题）

### Q23：ReAct 框架的流程是什么？Agent 如何实现"思考-行动-观察"循环？

> **答案要点**：ReAct 循环：Thought（当前推理）→ Action（调用工具/回答）→ Observation（工具返回结果）→ 回到 Thought。Agent 将历史所有 Thought/Action/Observation 拼接为上下文，持续多轮直到得出最终答案或达到最大轮次。

### Q24：Function Calling（Tool Calling）的完整技术流程是什么？

> **答案要点**：
> ① 定义 Tool Schema（name + description + parameters JSON Schema）
> ② 将 Tool Schema 注入 System Prompt
> ③ LLM 输出结构化 Tool Call（通常格式：{"name": "...", "arguments": {...}}）
> ④ 解析调用请求 → 执行工具函数 → 返回结果
> ⑤ 将工具结果注入下一轮 LLM 调用
> 关键难点：Schema 准确性、参数校验、错误处理。

### Q25：LangGraph 的 StateGraph 机制是如何实现 Agent 工作流的？

> **答案要点**：StateGraph 将 Agent 抽象为：节点（Node）→ 执行逻辑；边（Edge）→ 状态转移；状态（State）→ 全局共享数据。通过条件边（Conditional Edge）实现 Thought → Action → Observation 的动态路由。Checkpointer 保存每一步状态，支持中断恢复。

### Q26：多 Agent 协作的核心模式有哪些？

> **答案要点**：
> ① **Supervisor/Worker 模式**：一个 Agent 分配任务，多个 Worker Agent 执行
> ② **Debate 模式**：多个 Agent 各自推理后辩论达成共识
> ③ **Pipeline 模式**：任务按阶段串行传递给不同 Agent
> ④ **Hierarchical 模式**：多层分级管理

### Q27：MCP（Model Context Protocol）的架构是怎样的？

> **答案要点**：MCP 采用 C/S 架构：MCP Client → MCP Server → 资源（Resources/Tools/Prompts）。Host（LLM 应用）通过 Client 与 Server 通信，Server 提供外部能力（数据库查询 / API 调用 / 文件操作）。Transport 层支持 stdio/SSE。

### Q28：Agent 如何解决长程任务中的错误积累（Error Accumulation）问题？

> **答案要点**：① Checkpoint & Rollback：关键步骤设置检查点，失败回退；② Reflection 机制：执行后让 Agent 自我检查纠错；③ 局部规划 vs 全局规划：频繁 replan；④ 验证节点：插入专门的 Verification Agent 检查中间结果；⑤ Human-in-the-Loop：高风险步骤人工确认。

### Q29：RAG 系统中 Chunking 策略如何选择？

> **答案要点**：
> | 策略 | 适用场景 | 优势/劣势 |
> |------|---------|-----------|
> | 固定分块（256/512 token） | 通用 | 简单但可能截断语义 |
> | 语义分块（Sentence Splitter） | 需要语义完整 | 质量高但效率略低 |
> | 递归分块 | 长文本 | 兼顾上下文 |
> | 特定格式分块（Markdown/代码） | 结构化内容 | 保持原有结构 |

### Q30：Reranking 在 RAG 中的作用是什么？如何实现？

> **答案要点**：第一轮检索（Embedding + 向量检索）取 Top-K（如 K=50），然后用更精细的 Cross-Encoder Reranker 重排取 Top-N（如 N=5）。作用：弥补 embedding 检索的精度不足，大幅提升最终检索质量。

### Q31：什么是 LLM as Judge？它有什么主要偏差？如何缓解？

> **答案要点**：用 LLM（如 GPT-4、Qwen）评估模型输出质量。主要偏差：① 位置偏差（偏好靠前的选项）；② 长度偏差（偏好更长的回答）；③ 自我增强偏差（偏好自己的输出风格）。缓解方法：① 多次交换选项位置取平均；② 使用专门的 Judge 模型；③ 提供详细评分标准。

### Q32：Self-RAG 的核心思想是什么？

> **答案要点**：Self-RAG 让模型在生成过程中自主决定是否需要检索（通过 Reflection Token），检索后基于检索结果生成片段。核心创新：模型自行控制检索时机，而不是"每个步骤都检索"或"只在开头检索一次"。

### Q33：Agent 的可观测性如何建设？主要观测维度有哪些？

> **答案要点**：
> 主要维度：① LLM 调用（推理耗时、token 消耗、返回质量）；② Tool 调用（调用成功率、执行时间、错误类型）；③ Agent 决策流（Thought 内容、路由路径）；④ 执行链路（Span Tree/Trace）。技术方案：OpenTelemetry + 自定义 Span + 日志聚合（如 Grafana/Loki）。

### Q34：Agent 安全方面有哪些关键机制？

> **答案要点**：① Tool 调用权限校验（最小权限原则）；② 沙箱环境隔离（避免 Agent 执行破坏性操作）；③ Guardrails（输入/输出内容过滤 + 敏感操作二次确认）；④ Kill Switch（紧急终止机制）；⑤ 可追溯与可审计（记录所有 Agent 操作日志）。

## 🅳 工程与系统设计（8 题）

### Q35：如何设计一个高可用的 LLM 推理服务？

> **答案要点**：
> ① 推理引擎选型（vLLM / TGI / TensorRT-LLM）
> ② 负载均衡（按模型/版本进行请求分发）
> ③ 批处理策略（Continuous Batching 提高吞吐）
> ④ 缓存策略（Prefix Cache + Response Cache）
> ⑤ 弹性伸缩（HPA 根据 GPU 利用率自动扩缩）
> ⑥ 降级方案（模型降级到小模型/规则兜底）

### Q36：ZeRO Stage 1/2/3 分别优化了什么？

> **答案要点**：
> - **ZeRO-1**：Shard 优化器状态（显存节省 4x）
> - **ZeRO-2**：Shard 优化器状态 + 梯度
> - **ZeRO-3**：Shard 优化器状态 + 梯度 + 模型参数
> ZeRO-3 可在 N 卡上训练 N 倍参数的模型，但通信开销最大。

### Q37：Redis 缓存穿透、缓存击穿、缓存雪崩的区别和解决方案？

> **答案要点**：
> - **穿透**：查询不存在的数据 → 布隆过滤器 + 空值缓存
> - **击穿**：热点 key 过期 → 互斥锁 / 逻辑过期（后台刷新）
> - **雪崩**：大量 key 同时过期 / Redis 宕机 → 过期时间随机化 + 多级缓存 + Redis 集群

### Q38：goroutine 和 channel 在 Go 中是如何工作的？

> **答案要点**：goroutine 是 Go 的轻量级线程（初始栈小，可动态增长），由 GMP 调度器管理。channel 是 CSP 模型的核心，用于 goroutine 间的通信（有缓冲/无缓冲）。核心原则："Don't communicate by sharing memory; share memory by communicating."

### Q39：设计一个可靠的 Agent Tool 调用系统需要考虑哪些方面？

> **答案要点**：① Schema 设计（明确的入参/出参定义 + 校验规则）；② 错误处理（超时/重试/Fallback）；③ 幂等性（避免重复执行造成副作用）；④ 权限控制（Tool 级别的访问控制）；⑤ 并发控制（避免竞态）；⑥ 审计日志（谁、何时、调用了什么、结果）。

### Q40：如何设计一个 Agent 评测的 A/B Testing 系统？

> **答案要点**：
> ① 实验组/对照组：旧版本 vs 新版本
> ② 关键指标（Key Metrics）：Task Success Rate、Tool Call Accuracy、Latency、Cost
> ③ 流量分配：按用户/请求 hash 一致性路由
> ④ 统计显著性检验：用 T-test / Mann-Whitney U 检验
> ⑤ 长期追踪：防止 novelty effect（霍桑效应）

### Q41：MySQL 索引失效的常见场景有哪些？

> **答案要点**：① 对索引列使用函数或计算；② 隐式类型转换；③ 左模糊查询 LIKE '%xxx'；④ 联合索引违反最左前缀原则；⑤ OR 条件（部分列无索引）；⑥ 数据分布不均匀（优化器认为全表扫描更快）。

### Q42：什么是 Service Mesh？在 AI 平台中如何应用？

> **答案要点**：Service Mesh（服务网格）将服务间通信抽离到基础设施层（Sidecar Proxy），提供服务发现、负载均衡、熔断、可观测性等能力。在 AI 平台中用于：各微服务（训练/推理/数据/标注/评测）间的可靠通信、流量管理、链路追踪。

## 🅴 Coding Agent 与 AI 辅助开发（4 题）

### Q43：Coding Agent 常见的 Failure Mode 有哪些？

> **答案要点**：① 上下文窗口溢出（长文件被截断导致错误修改）；② 幻觉引入不存在的 API/库（"import" 一个不存在库）；③ 循环调试陷阱（反复尝试同一错误方案）；④ 忽略项目已有模式（不遵循代码风格/架构）；⑤ 过度删除（删了不该删的代码）。

### Q44：如何评价一个 Coding Agent 的效果？

> **答案要点**：评价维度：① Pass@k（生成代码的通过率）；② 编辑准确率（正确修改 vs 错误修改）；③ 任务完成率（端到端的功能实现）；④ 回归率（有无破坏已有功能）；⑤ 效率提升（人工完成 vs AI 辅助的时间对比）。

### Q45：Agentic Coding 的"能力边界"是什么？如何设计安全护栏？

> **答案要点**：能力边界：① 能做好局部修改但难以全局重构；② 理解不了深层业务语义；③ 容易被过时文档或代码误导。安全护栏：① 关键操作需人工确认；② 限制 Agent 可以操作的文件范围；③ 自动测试验证每次修改；④ 提交前 Code Review。

### Q46：OpenCode / Cline 的配置和工作原理是什么？

> **答案要点**：Agent 配置文件通常定义了：可使用的工具集（Tool Set）、权限范围、模型选择、上下文长度等。工作模式：用户指令 → Agent 规划（Plan）→ 工具调用（Read/Write/Bash/Search）→ 输出结果。通过 yaml/json 自定义技能（Skills）和规则。
